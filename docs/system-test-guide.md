# 系統測試說明書

這份說明書用來重新檢查目前前端原型。測試重點不是證明系統可以真實派工，而是確認使用者能看懂：資料仍是未整理資料、草稿仍需要人工確認、行動端只能標記查看候選草稿。

## 測試前準備

1. 在專案根目錄執行：

```bash
pnpm install
pnpm dev
```

2. 打開本機網址，從首頁開始測試。
3. 測試時不要輸入真實姓名、地址、電話、真實餐廳或真實災情。
4. 測試後執行：

```bash
pnpm format
pnpm test
pnpm build
pnpm check
```

## 測試角色

請至少用以下 5 種視角各走一次流程：

| 角色       | 主要測試重點                                   |
| ---------- | ---------------------------------------------- |
| 資料整理者 | 是否能看見原文、來源、查核狀態與不能行動原因。 |
| 回報者     | 回報便當需求時，是否知道這不是正式訂單。       |
| 行動志工   | 標記查看草稿時，是否知道這不是正式派工。       |
| 志願餐廳   | 供應量與餐廳狀態是否不會被誤解成承諾訂單。     |
| 政府協調者 | 系統是否清楚標示責任邊界與人工確認需求。       |

## 本輪使用者回饋採用摘要

5 位 sub-agent 使用者共同指出：原本的「Submit 到行動端」「任務領取系統」「領取任務草稿」語氣太像正式派工；餐廳供應數字和預設勾選也容易被誤解成已承諾或已確認。

本輪採用的調整：

- 將「Submit 到行動端」改成「送出候選草稿供討論」。
- 將「任務領取系統」改成「候選草稿檢視」。
- 將「領取任務草稿」改成「標記已查看草稿」或「標記願意討論供應」。
- 將 `TASK-*` 編號改成 `CAND-*`，避免正式任務語感。
- 新增責任邊界區塊，明確列出不判定真偽、不核准任務、不派工、不下單、不付款、不通知餐廳。
- 將餐廳供應表改成「示範可討論量」「分配假設草稿」「示範取餐時段」。
- 將分配檢核改成待確認語氣，避免預設勾選造成已確認印象。

## 手動測試流程

### 1. 首頁與三端切換

測試步驟：

1. 打開首頁。
2. 確認看得到「資料整理端」「回報端」「行動端」三個入口。
3. 點選三個入口，確認畫面會切換。

應看到：

- 首頁標示這是「災害資訊協作原型」。
- 畫面說明不接收真實現場資料、地址、電話或個資。
- 每一端都有自己的用途說明。

不可出現：

- 暗示這是正式救災系統。
- 暗示資料已完成查證。
- 暗示可以直接派工、下單、付款或配送。

### 2. 資料整理端儀表板

測試步驟：

1. 停在「資料整理端」。
2. 查看「整理儀表板」。
3. 點「查看缺口」。
4. 點不同資料卡，確認選取資料會更新。

應看到：

- 查核狀態包含「待人工確認」或「未查核」。
- 資料風險熱度圖說明這不是救災優先順序。
- 缺口檢視會顯示目前不能直接使用的原因。

不可出現：

- 把 `needs_review` 或 `unverified` 顯示成已確認。
- 用熱度圖暗示救災優先順序。
- 只顯示整理結果，卻看不到原文或缺口。

### 3. 進入整理草稿區

測試步驟：

1. 點「整理草稿區」。
2. 確認出現拼圖校準閱讀提醒。
3. 在未完成拼圖前，確認「進入整理草稿區」不能按。
4. 將拼圖滑到右側完成校準。
5. 進入整理草稿區。

應看到：

- 提醒資料仍是未整理資料。
- 提醒拼圖校準只是在本機確認使用意圖。
- 進入後仍看得到不能直接變成任務的原因。

不可出現：

- 把拼圖校準說成外部驗證。
- 把拼圖校準說成資料已查證。
- 進入後變成正式派工畫面。

### 4. 草稿整理與送到行動端

測試步驟：

1. 在整理草稿區選一筆資料。
2. 查看原文、判斷追溯、流程檢核與草稿編輯。
3. 在「送出候選草稿供討論」區塊先不要選角色，確認按鈕不能按。
4. 用角色下拉選擇「資料整理者」。
5. 按下「送出候選草稿供討論」。

應看到：

- 送出前需要角色下拉選取。
- 送出後自動切到行動端。
- 送出後候選草稿狀態顯示紅色「實行專案」。
- 提示文字說明這仍是候選任務草稿，需要人工確認。

不可出現：

- 不選身份也能送出。
- 送出後顯示為正式任務。
- 把紅色「實行專案」解釋成已核准或已派工。
- 送出後移除「不可直接行動」或「需要人工確認」的邊界。

### 5. 回報端便當需求

測試步驟：

1. 切到「回報端」。
2. 選擇誰需要便當、需要數量與餐別。
3. 按「暫存成待確認草稿」。
4. 回到「資料整理端」，確認新增的 `BENTO-*` 草稿可被檢查。

應看到：

- 回報端只記錄群組或角色。
- 不要求真實姓名、電話或地址。
- 送出後是資料整理端草稿，不是正式訂單。

不可出現：

- 「付款」「已通知餐廳」「訂單成立」等語意。
- 真實餐廳、真實地址、真實電話。
- 直接把便當需求放進行動端正式任務。

### 6. 行動端候選草稿檢視

測試步驟：

1. 切到「行動端」。
2. 若沒有任務，先回資料整理端送出一筆草稿。
3. 在行動端選擇查看角色，例如「志工」。
4. 勾選「我知道這只是候選草稿」。
5. 按「標記已查看草稿」。

應看到：

- 任務狀態是草稿或候選。
- 標記查看後仍顯示不是正式派工。
- 任務卡保留卡住點或不可直接行動原因。

不可出現：

- 「已派工」「已完成」「可立即執行」等正式行動語意。
- 標記查看後移除安全邊界。
- 將查看角色當成真實驗證或官方授權。

### 7. 行動端餐廳供應分配

測試步驟：

1. 在「行動端」查看餐廳供應表。
2. 確認每間餐廳有示範可討論量、分配假設草稿、仍可討論、餐別、類型、示範取餐時段與確認狀態。
3. 查看每列限制與建議動作。
4. 查看分配前待確認。

應看到：

- 餐廳名稱都是示範名稱。
- 供應量仍是 mock 草稿。
- 分配假設草稿不等於訂單成立。

不可出現：

- 真實餐廳承諾。
- 真實付款或配送流程。
- 把備援或待確認狀態寫成已接受訂單。

## 自動化測試檢核

自動化測試目前至少要涵蓋：

- 首頁標題與安全邊界。
- 三端切換。
- 回報端便當需求進入資料整理端草稿。
- 拼圖校準後才能進入整理草稿區。
- 草稿送到行動端後可以被標記查看。
- 不出現已移除的預覽、派工或正式任務入口。

執行：

```bash
pnpm test
pnpm check
```

## 回饋紀錄格式

每次測試後，用這個格式記錄：

| 角色 | 測試流程 | 看到的問題 | 風險 | 建議調整 | 是否採用 |
| ---- | -------- | ---------- | ---- | -------- | -------- |
|      |          |            |      |          |          |

採用建議前請先判斷：

- 是否會讓未確認資料看起來更像已確認。
- 是否會讓草稿看起來像正式任務。
- 是否需要真實資料、後端、資料庫、外部 API 或地圖。
- 是否仍符合目前前端-only 的課程範圍。
