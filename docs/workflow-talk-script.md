# Workflow 講解稿

這份講解稿可以搭配 `docs/flow.md` 的 Mermaid 圖使用。建議時間約 3 到 5 分鐘。

## 30 秒版

我們這次的 workflow 不是正式救災流程，而是資訊整理流程。核心是：原始資訊或便當需求先進到資料整理端，整理者檢查來源、原文、查核狀態和缺口，再決定是否形成候選草稿。候選草稿可以送到行動端示意給下一位協作者查看，但只能標記已查看，不能派工、下單、付款或通知餐廳。

## 3 分鐘版

### 1. 開場

我們的系統定位是「災害資訊協作原型」，不是正式救災系統。這點很重要，因為 Phase 0 的資料都是未整理資料，很多內容可能是轉述、模糊、重複或還沒查證。

所以這個 workflow 的目的不是把資訊變成任務，而是讓資訊在變成任務以前先停下來，被資料整理者檢查。

### 2. 三個端點

畫面上有三個端點：資料整理端、回報端、行動端示意。

資料整理端是主流程。它負責看原文、看資訊取得方式、看查核狀態，並留下「為什麼不能直接行動」的理由。

回報端只是示範便當需求如何進來。使用者可以選群組、數量和餐別，但送出後只會暫存成待確認草稿，不會真的下單，也不會通知餐廳。

行動端示意只用來看候選草稿。它不是派工系統，所以我們把原本比較像正式行動的詞改掉，例如不說「領取任務」，而是說「標記已查看草稿」。

### 3. 主要 Workflow

第一步是原始資訊進入資料整理端。整理者先看來源、原文、更新時間和查核狀態。

接著會做三個檢查：

第一，是否可能重複上傳。如果可能重複，系統不會自動合併，只會標示需要人工確認。

第二，來源是否可追溯。如果來源不清楚，例如截圖沒有日期或轉傳沒有原始發布者，就暫存為可疑來源，不能當成已確認。

第三，訊息是否清楚。如果缺少時間、地點、數量、當事人或負責窗口，就留下補問問題。

只有在整理者保留原文、卡住點和不可直接行動原因之後，這筆資料才可以變成候選草稿。

### 4. 候選草稿送到行動端

候選草稿可以送到行動端示意，但送出前要選角色聲明。這個下拉選單不是身分驗證，也不是授權，只是讓我們在原型中模擬不同角色會怎麼看這份草稿。

送到行動端後，查看者還要勾選確認：這不是正式派工、不是餐廳承諾、不是訂單，也不是付款。

送出後的候選草稿會用紅色顯示「實行專案」，這是為了讓狀態醒目；它仍然只是行動端草稿狀態，不代表已核准、已派工或已確認。

如果查看者是志工，他只能標記已查看草稿。

如果查看者是志願餐廳，他只能標記願意討論供應，不代表已接受訂單。

如果查看者是政府或協調單位，他看到的是風險與責任邊界，不代表官方決策。

### 5. 責任邊界

整個 workflow 最重要的是責任邊界。

這個系統不判定真偽、不核准任務、不派工、不下單、不付款，也不通知餐廳。

所以畫面上的 `BENTO-*` 是便當需求草稿，不是訂單編號；`CAND-*` 是候選草稿，不是任務編號。

我們希望下一位協作者看到的不是「這件事可以做」，而是「這件事現在為什麼還不能直接做，以及需要補確認什麼」。

### 6. 收尾

總結來說，這個 workflow 把資訊分成三個階段：進來、整理、示意查看。

每個階段都保留不確定性，避免原始資訊在轉換過程中被包裝成已確認任務。這也是我們這版 prototype 最想展示的核心價值。

## 可能被問到的問題

### 為什麼要有回報端？

回報端是用來示範資訊入口，不是正式回報網站。它讓我們看到，如果一個便當需求被填進來，應該先變成待確認草稿，而不是直接下單。

### 為什麼要有行動端？

行動端是用來示範資訊被下一位協作者看到時，可能產生的誤解。所以我們刻意把它設計成「候選草稿檢視」，而不是正式派工。

### 為什麼不直接叫任務？

因為目前資料仍需要人工確認。叫任務會讓使用者以為可以直接執行，這正是我們要避免的風險。

### 角色下拉是不是身分驗證？

不是。角色下拉只是本機原型中的角色聲明，沒有外部驗證、沒有帳號權限，也不代表官方授權。

### 餐廳數量是真的嗎？

不是。餐廳名稱、可討論量、分配假設草稿和取餐時段都是 mock 示範資料，不代表真實餐廳承諾。

## 10 秒結論

這個 workflow 的核心不是「派出去」，而是「先停下來」。先讓資訊整理者看見不確定性，再讓下一位協作者知道哪些地方還不能直接行動。
